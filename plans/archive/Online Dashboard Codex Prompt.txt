5. API/proxy requirements
Build a small API/proxy from the start.

Recommended platform:
- Cloudflare Workers

Recommended route shape:
- /api/*

Recommended deployment shape:
- GitHub Pages serves /
- Cloudflare routes /api/* to the Worker

API responsibilities:
- Hold real secrets server-side
- Query live data sources such as Supabase / database / existing operational sources
- Query historical/manifests data such as R2 indexes/manifests/history metadata
- Return only the JSON needed by the dashboard
- Normalise upstream data as needed
- Keep front-end data shape aligned with the local dashboard as much as possible

Important:
- Do not let the browser fetch private R2 or secret-bearing endpoints directly
- All live and historical data access should go through the API/proxy in V1

6. Security rules
Browser code must never receive:
- Supabase service role key
- R2 private credentials
- Cloudflare API tokens
- database admin credentials
- any secret that can write, delete, or bypass auth

Browser code may receive only:
- browser-safe config
- API base URL
- non-sensitive display/config values

Treat the browser as untrusted even though access is behind Zero Trust.

8. API route recommendations
Implement a thin but clean route structure. Adjust naming only if needed to match the existing local dashboard expectations.

Recommended routes:
- GET /api/health
- GET /api/status/summary
- GET /api/status/feeds
- GET /api/status/db
- GET /api/status/history
- GET /api/history/manifests
- GET /api/history/runs

Route purposes:
- /api/health
  - simple uptime/health response for the API itself
- /api/status/summary
  - top-level summary for key dashboard cards
- /api/status/feeds
  - feed/source operational status
- /api/status/db
  - DB usage/storage/size metrics needed by the dashboard
- /api/status/history
  - history/manifests/R2 health summary
- /api/history/manifests
  - structured manifest/index data for history panels
- /api/history/runs
  - ingestion/backfill/run metadata if the local dashboard uses this

Use a consistent JSON envelope:
Success:
{
  "ok": true,
  "generatedAt": "2026-04-08T12:00:00Z",
  "data": { ... }
}

Failure:
{
  "ok": false,
  "generatedAt": "2026-04-08T12:00:00Z",
  "error": {
    "code": "UPSTREAM_TIMEOUT",
    "message": "Timed out fetching manifest index"
  }
}

Even if the front end does not yet surface these errors nicely, keep the API structured properly.
