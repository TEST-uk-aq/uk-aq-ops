### Browser instrumentation requirements

- Ensure each chart load attempt emits at most one telemetry payload.
- Send telemetry after completion and also on failure paths.
- Prefer `navigator.sendBeacon()` for delivery.
- Add a fallback using `fetch(..., { keepalive: true })`.
- Do not block chart rendering on telemetry.
- Do not let telemetry errors break the chart.
- Keep payload compact.
- Structure code so sampling can be introduced later with one obvious gate, but do not sample yet.

### Cache classification rules
Implement cache classification clearly and centrally so future changes are easy:
- `obs_used_local_cache`: true when observations were available from local/browser cache before or instead of a network result
- `aqi_used_local_cache`: same idea for AQI
- `obs_received_304`: true when observation fetch was validated with ETag and got 304
- `cache_session_was_warm`: true when the existing cache/session warm-up was already satisfied
- `overall_cache_class`:
  - `warm_local` when local/browser cached data materially served the load and there was no obvious cold fetch dominance
  - `warm_http_304` when HTTP validation / 304 materially served the load
  - `mixed` when both cache-assisted and network fetch behaviour are involved
  - `cold` when it is effectively a fresh network-driven load
  - `bypass` if a no-cache / forced-refresh style path exists
  - `unknown` as safe fallback

Use pragmatic logic and add comments documenting the exact rules.
