# Prune Daily PM DAQI fix

**Date:** 17 July 2026  
**Target environment:** TEST only  
**Primary repository:** `TEST-uk-aq/uk-aq-ops`  
**Related repository if a new ObsAQIDB RPC is required:** `TEST-uk-aq/uk-aq-schema`  
**Status:** Proposed implementation plan

## 1. Purpose

Correct the permanent R2 AQI generated by Prune Daily so PM2.5 and PM10 DAQI values use the preceding 23 hourly observation buckets when calculating each target day's rolling 24-hour mean.

The defect is visible in the R2 AQI for 12 July 2026:

- PM EAQI is present because it uses the target hour's hourly mean;
- PM DAQI is missing for much or all of the day because it requires 24 hourly inputs;
- the AQI Parquet files, manifests and indexes were written successfully, so this is a calculation-context defect rather than an R2 write failure.

This plan is for the TEST system only. Do not contact, deploy to, query destructively, or modify LIVE.

## 2. Current-code diagnosis

The current Phase B implementation already attempts to include prior-day context, but it obtains that context from the wrong database for the point in the retention cycle when Phase B runs.

In `workers/uk_aq_prune_daily/phase_b_history_r2.mjs`:

1. `stageFrozenSourceAndWriteObservations()` calculates a source window from the start of D-1 to the end of D.
2. It queries `uk_aq_ops.uk_aq_phase_b_history_rows_v2(...)` through `SUPABASE_DB_URL`, which is the IngestDB PostgreSQL connection.
3. It stores the result in the bounded local frozen source.
4. Only rows belonging to target day D are written to the R2 observations partition.
5. All supported AQI rows in the frozen source are passed to `buildAqilevelHistoryRowsForDayFromSourceObservations(...)`.
6. The shared AQI library correctly looks back 23 hours and requires 24 hourly PM values before returning a PM DAQI index.

The calculation logic is therefore already correct. The problem is source availability:

- Phase B processes a day after it has moved beyond the configured IngestDB retention window.
- At that point, D can still be available for the Phase B candidate, but D-1 has already been removed from IngestDB.
- The D-1 part of the existing query therefore returns no rows or insufficient rows.
- The PM rolling calculation starts again at 00:00 on D and reports `insufficient_samples` until 24 target-day hours have accumulated.
- EAQI remains available because it only requires the current hourly mean.

The correction is not to add another 23-hour lookback to the shared AQI algorithm. It is to source the existing 23-hour lookback from ObsAQIDB.

## 3. Important scale constraint

Do not simply fetch the entire previous day of raw observations from ObsAQIDB and append it to the current frozen NDJSON file.

The 12 July run included a connector/day with approximately 228,810 target-day observations while the configured Phase B frozen-source cap was 250,000 rows. A full extra day of raw observations would exceed the current safety cap and could greatly increase temporary storage, memory use and processing time.

The context source should therefore be narrow and hourly-aggregated:

- PM2.5 and PM10 only;
- one connector at a time;
- exactly the preceding 23 UTC hours needed for target day D;
- hourly mean and sample count rather than every raw observation;
- bounded and paginated;
- calculation-only, never written into the target day's R2 observation partition.

## 4. Required end state

For a target day D:

```text
Target-day observation source
  IngestDB frozen source for D 00:00 inclusive to D+1 00:00 exclusive

PM rolling context source
  ObsAQIDB hourly PM context for D-1 01:00 inclusive to D 00:00 exclusive

AQI calculation input
  23 context hours from ObsAQIDB
  + target-day hourly values derived from the frozen IngestDB source

R2 observation output
  target-day observations only

R2 AQI output
  target-day AQI rows only
```

For an output hour H, PM DAQI must use:

```text
H - 23 hours through H
```

NO2 behaviour must not change. Both NO2 DAQI and NO2 EAQI continue to use the target hour's hourly mean and do not need prior-day context.

## 5. Fixed implementation decisions

These are requirements, not matters for redesign during implementation.

### 5.1 Keep the shared AQI algorithm authoritative

Continue to use `lib/aqi/aqi_levels.mjs` for:

- pollutant normalisation;
- raw observation hourly aggregation;
- PM rolling 24-hour calculations;
- DAQI and EAQI breakpoints;
- required sample counts;
- calculation statuses and missing reasons;
- algorithm version.

Do not copy breakpoints or recreate rolling calculations in Prune Daily or SQL.

### 5.2 Keep target-day observations frozen from IngestDB

The same target-day frozen source must continue to feed:

- the target-day R2 observation write;
- the target-day part of the AQI calculation.

ObsAQIDB is only the source of the preceding PM calculation context. It must not replace the target-day frozen source or alter observation-history row counts, manifests or candidate completion checks.

### 5.3 Fetch hourly context, not raw prior-day rows

Add a narrow service-role ObsAQIDB RPC, unless an existing deployed RPC already provides the exact bounded contract.

Recommended contract:

```text
uk_aq_rpc_observs_aqi_pm_hourly_context
```

Inputs:

```text
p_connector_id
p_start_utc
p_end_utc
p_after_timeseries_id
p_after_timestamp_hour_utc
p_limit
```

Output ordered by `timeseries_id, timestamp_hour_utc`:

```text
connector_id
station_id
timeseries_id
pollutant_code
timestamp_hour_utc
hourly_mean_ugm3
sample_count
```

The RPC must:

- require `service_role`;
- query ObsAQIDB observations;
- join authoritative metadata to identify `pm25` and `pm10`;
- reject invalid connector IDs or time windows;
- reject a window longer than 24 hours;
- use keyset pagination;
- cap `p_limit` to a safe maximum;
- return only PM2.5 and PM10;
- use the same hourly bucket boundary as the shared JavaScript AQI library;
- return a successful empty result when no context observations genuinely exist.

Do not expose a broad public raw-observation query for this change.

### 5.4 Target-day frozen source wins any overlap

The intended context window ends at D 00:00 and should not overlap target-day rows. Nevertheless, merge defensively by canonical hourly key:

```text
timeseries_id + pollutant_code + timestamp_hour_utc
```

If an overlap occurs, the hourly value derived from the target-day frozen IngestDB source must win over the ObsAQIDB context value.

### 5.5 Context is not output

Context rows must never:

- be written to `history/v2/observations/day_utc=D`;
- increase the target-day observation candidate row count;
- appear as AQI output rows before D 00:00;
- affect target-day observation manifests or checkpoints;
- satisfy a target-day source requirement when D contains no supported AQI observations.

### 5.6 Fail closed on an incomplete context read

A successful complete query returning zero rows is legitimate. It should produce the existing `insufficient_samples` result where applicable.

The candidate must fail and pruning must remain blocked when:

- the ObsAQIDB context request fails;
- pagination is truncated or exceeds its configured cap;
- the response cannot be normalised safely;
- the required context window is outside the configured ObsAQIDB retention period;
- the service cannot distinguish a complete empty result from an incomplete read.

Do not silently calculate apparently authoritative R2 PM DAQI from a partially fetched context window.

### 7.2 Separate target-day source from PM context

In `workers/uk_aq_prune_daily/phase_b_history_r2.mjs`:

1. Change `stageFrozenSourceAndWriteObservations()` so its IngestDB extraction covers target day D only:


5. Build the set of target-day PM timeseries that can require context.
6. Fetch the preceding 23 hourly PM context rows from ObsAQIDB through the narrow RPC.
7. Discard context for timeseries or pollutants that do not occur in the target-day PM source.
8. Convert target-day raw observations to the shared library's hourly narrow-row form.
9. Merge the ObsAQIDB hourly context and target-day hourly rows with target-day precedence.

### 7.4 Retention guard

At Phase B run start, calculate the earliest required PM context timestamp for each candidate and compare it with the configured ObsAQIDB observation-retention boundary.

Do not rely on a brittle hard-coded relationship between the two retention-day variables. Use actual UTC timestamps and the same retention semantics as the current cleanup code.

For a normal candidate the context should be available because ObsAQIDB retains observations longer than IngestDB. For an old pending candidate outside ObsAQIDB retention:

- fail that candidate closed with a clear reason;
- leave pruning blocked;
- direct remediation to the R2 observation-based integrity/backfill path.
