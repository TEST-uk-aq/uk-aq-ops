Yes, `local_to_aqilevels` exists, but it is **not the right mode for “local Dropbox observations → calculate AQI → write AQI history to R2”**.

`local_to_aqilevels` is badly named for what you want. In the active `run_job.ts`, it:

1. Requires **Ingest Supabase** credentials and **ObsAQIDB Supabase** credentials.
2. Checks connector counts from `ingestdb` and `obs_aqidb`.
3. Can read R2 observation history as a source if it chooses `sourceKind === "r2"`.
4. But then it calls `upsertAqilevelsRows(...)`, which writes to **ObsAQIDB**, not R2.

So `local_to_aqilevels` is basically:

```text id="y99ad0"
source rows / maybe R2 observations → calculate AQI helper rows → write ObsAQIDB hourly/rollups
```

not:

```text id="4atp3q"
Dropbox R2 mirror observations → calculate AQI parquet/manifests → upload aqilevels to R2
```

## The mode that writes AQI history to R2

The R2-writing mode is:

```text id="lvnv58"
r2_history_obs_to_aqilevels
```

with:

```text id="9d2veh"
UK_AQ_BACKFILL_OUTPUT_SCOPE=aqilevels_only
```

The docs say `aqilevels_only` rebuilds AQI history outputs from committed R2 observation history, and is valid only with `r2_history_obs_to_aqilevels`.  The wrapper also says this mode “reads committed R2 observations and rewrites AQI history” for v1 or v2, and for v2 uses the v2 observations and AQI data/debug prefixes.

That is the active supported route for rebuilding 19/06 AQI in R2.

## But your exact “local Dropbox files → write R2” path is different

## Bottom line

`local_to_aqilevels` is for writing calculated AQI into **ObsAQIDB**, not R2. For your immediate need, use `r2_history_obs_to_aqilevels` with `aqilevels_only`. For a local Dropbox-based check first, use `uk_aq_validate_aqi_from_dropbox_observs.mjs`.

The system doc should probably be corrected, because it lists `local_to_aqilevels` but does not explain that it upserts to ObsAQIDB, and the `aqilevels_only` section still describes the v1 path even though the wrapper supports v2.
