# Latest snapshot behavioural contract

## Authority

This file is the authoritative behavioural contract for the latest-snapshot system.

Implementation files, worker-local READMEs, plans and archives MUST conform to this contract unless an intentional contract change is approved and documented in the same branch or pull request.

## Purpose

The latest-snapshot system provides deterministic, cacheable website data representing the latest valid pollutant observation for each eligible timeseries.

It is a current-state product. It is not the authoritative raw observation record.

## Scope

### Public request matrix

- pollutants: `pm25`, `pm10`, `no2`;
- windows: `3h`, `6h`, `1d`, `7d`, `all`;
- network group: `all`;
- output contract: `v2`.

### Physical R2 matrix

- pollutants: `pm25`, `pm10`, `no2`;
- physical window: `all` only;
- network group: `all`;
- physical object count: three.

The public request matrix and physical R2 matrix are intentionally different. Finite windows are derived by the R2 API Worker and are not separately stored products.

## Definitions

### Raw observation

A source observation received by the wider UK AQ ingestion system. Raw observations may include sentinel, invalid, outlier or otherwise unusable values and remain part of source history.

### Valid current pollutant value

A value eligible to become the latest public current value for a pollutant timeseries.

For the current matrix, the value MUST:

1. be numeric and finite;
2. be greater than or equal to `0`;
3. satisfy any existing pollutant-specific upper bound.

The current upper-bound behaviour is:

- PM2.5: maximum `500`;
- PM10: maximum `600`;
- NO2: no additional upper bound in this component.

Zero is valid. A negative pollutant value, including `-99`, is not valid current state.

### Latest valid state

The single retained latest valid observation for one `(connector_id, timeseries_id)` identity.

### Physical snapshot

The stable, hash-gated R2 object generated by the Cloud Run builder for one pollutant with `window=all`.

### Finite public response

A request-time representation produced by filtering a physical `all` snapshot to `3h`, `6h`, `1d` or `7d` using each row's `last_value_at`.

## Required invariants

### Raw-data separation

- The observation publisher MUST continue publishing raw observations needed by raw observation consumers.
- The latest-snapshot system MUST use its dedicated Pub/Sub subscription and MUST NOT consume the raw writer's subscription.
- Latest-current-value eligibility MUST NOT prevent invalid or sentinel rows from reaching raw `observations`, `observs` or R2 observation history.
- Invalid current values MUST be rejected by this consumer, not removed from the shared raw stream.

### State identity and replacement

- State identity MUST remain `(connector_id, timeseries_id)`.
- A valid observation MUST create state when no state exists.
- A newer valid observation MUST replace older valid state.
- An older valid observation MUST NOT replace newer valid state.
- An invalid observation MUST NOT create or replace state.
- The previous valid state row MUST remain unchanged until a newer valid observation arrives.
- Existing same-timestamp valid-row tie-breaking using `ingested_at` MUST remain unchanged unless separately approved.

### Message acknowledgement

- Malformed messages MAY be acknowledged as malformed according to existing behaviour.
- Decoded valid and invalid observation messages MUST be acknowledged after successful state handling.
- A deliberately skipped invalid value is successfully handled and MUST NOT be left for endless redelivery.
- If state persistence fails, acknowledgement MUST NOT move ahead of the failed state handling.

### State persistence

- State MUST remain at `latest_snapshots_state/v1/latest_state.json` unless an approved migration changes it.
- The state schema remains version `1`.
- State MUST contain only entries eligible to act as latest current values for the supported pollutant matrix.
- State writes SHOULD remain hash-gated.

### Metadata eligibility

- Timeseries metadata must resolve through the core metadata cache.
- The timeseries must resolve to its station, connector, phenomenon and observed property as required by the v2 row contract.
- Public network identity must resolve through `station.network_id -> networks.id`.
- Missing required station or network metadata is counted and skipped.
- Networks with `public_display_enabled=false` are skipped.
- The existing station geography requirement remains in force.
- Connector-derived network fallbacks MUST NOT be introduced.

### Observation time and window behaviour

- Persisted state uses `observed_at` as the retained valid observation timestamp.
- Public rows expose that timestamp as `last_value_at`.
- A newer invalid observation MUST NOT refresh the retained timestamp.
- The physical `window=all` object includes every metadata-eligible retained valid state row regardless of age.
- Finite cutoffs MUST use the start of the current UTC minute.
- A finite row is included when its parseable `last_value_at` is greater than or equal to the cutoff.
- A row with missing or unparseable `last_value_at` MUST be excluded from finite responses.
- Filtering MUST preserve the physical object's row order.

### Physical snapshot generation

The builder MUST:

- publish only `window=all` for each configured supported pollutant;
- use deterministic keys for `(network_group, pollutant, all)`;
- preserve the v2 top-level and row fields;
- sort each pollutant's rows using the existing ordering;
- derive the existing cursor for the physical payload;
- use stable JSON and SHA-256 hash gating;
- skip unchanged object writes;
- preserve a previous `all` manifest entry when that pollutant build fails and an entry exists.

`UK_AQ_LATEST_SNAPSHOT_WINDOWS` is not a builder configuration input. Public finite windows are owned by the R2 API Worker.

### Manifest

The manifest MUST describe physical stored products only.

For the current system:

- `matrix.pollutants` contains `pm25`, `pm10` and `no2`;
- `matrix.windows` contains only `all`;
- a fully successful build contains three snapshot entries;
- each entry points to a physical `window=all.json` object.

The manifest MUST NOT invent virtual entries for finite API responses.

### R2 API Worker

For every accepted latest-snapshot request, the Worker MUST read the requested pollutant's physical `window=all` object.

For `window=all`, it MUST return the stored representation directly with the physical object ETag.

For a finite window, it MUST:

1. parse and validate the physical v2 payload;
2. apply the UTC-minute cutoff to `last_value_at`;
3. preserve row order;
4. set `window` to the requested label;
5. recalculate `count`;
6. recalculate `next_since` and `next_since_id` using the existing cursor meaning;
7. preserve all other top-level and row fields;
8. return `X-UK-AQ-Snapshot-Contract: v2`.

The Worker MUST fail closed on malformed physical payloads and MUST NOT fall back to old finite objects or v1.

### Finite response cache identity

A finite response can change solely because time passes.

Its ETag MUST therefore depend on:

- the physical source object ETag;
- the requested finite window;
- the effective UTC minute.

Matching `If-None-Match` requests MAY return `304`. The existing cache-control configuration remains in force.

### Public v2 compatibility

The current architecture MUST preserve:

- the `latest_snapshots/v2` prefix;
- the manifest path;
- query parameter names and accepted public values;
- the cache-proxy route;
- response field names and meanings;
- scalar network fields;
- connector provenance fields;
- `X-UK-AQ-Snapshot-Contract: v2`;
- fail-closed behaviour for non-v2 standard paths.

## Old finite objects

Old `window=3h.json`, `window=6h.json`, `window=1d.json` and `window=7d.json` objects may remain in R2 after cutover.

They are inert historical artefacts. They MUST NOT be updated, advertised in the current manifest, or used as fallbacks.

## Explicit non-goals

A Latest Snapshot change MUST NOT, unless separately approved:

- delete or rewrite raw observation history;
- change AQI or WHO calculations;
- change connector checkpoint semantics in another repository;
- add pollutants or public window labels;
- change station display-name formatting;
- change network assignment or visibility rules;
- change cache-proxy routes or authentication;
- change Cloud Scheduler cadence, service resources, concurrency or timeouts;
- refactor unrelated shared R2 code.

## Implementation status

The latest-valid state ordering and the all-only physical snapshot architecture are implemented and running on TEST as of 17 July 2026.

## Contract-change rule

Changes to value eligibility, state identity, replacement ordering, physical product ownership, finite-window semantics, v2 fields, cache identity or failure behaviour require:

1. an update to this contract;
2. an update or new decision record;
3. only the smallest structural or targeted deterministic check genuinely needed;
4. functional validation through normal TEST operation.
