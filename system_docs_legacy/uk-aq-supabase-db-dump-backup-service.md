# uk_aq Supabase DB dump backup service

This document describes the production Cloud Run Service that performs daily logical Supabase database backups for:

- `ingestdb`
- `obs_aqidb`

The required chain is:

- Cloud Scheduler
- private authenticated Cloud Run Service
- Supabase CLI dump generation
- gzip compression
- Dropbox upload
- 7-day Dropbox retention cleanup

## Final architecture

1. Cloud Scheduler calls `POST /run-backup` daily at `00:55 UTC`.
2. The Cloud Run service accepts only authenticated callers and is deployed with internal ingress.
3. The service runs the two databases sequentially:
   1. `ingestdb`
   2. `obs_aqidb`
4. For each database it generates:
   - `roles.sql.gz`
   - `schema.sql.gz`
   - `data.sql.gz`
   - `cron_jobs.sql.gz`
5. Files are uploaded to Dropbox under dated folders.
6. Dropbox folders older than the retention window are deleted.
7. The service returns structured JSON and uses non-2xx status on failure.

## Important Supabase CLI implementation note

Official Supabase CLI docs describe `supabase db dump` as a container-driven `pg_dump` workflow.
Cloud Run Service is not a good place for Docker-in-Docker.

To stay compatible with Cloud Run while still using Supabase CLI dump logic, this implementation does the following:

1. Runs `supabase db dump --dry-run --db-url ...`
2. Extracts the exact bash script emitted by Supabase CLI
3. Executes that script locally inside the container using PostgreSQL client 17

This keeps the dump flags and filtering generated by Supabase CLI while avoiding Docker-in-Docker inside Cloud Run.

## Database configuration model

This implementation uses independent direct database connection strings, one per database:

- `UK_AQ_INGESTDB_DB_URL`
- `UK_AQ_OBS_AQIDB_DB_URL`

No Supabase project linking is required at runtime.
No Supabase access token is required when using the direct DB URL mode.

Each DB URL should be stored in Secret Manager and injected into Cloud Run as an env secret.

## Dropbox layout

Base root:

- `{UK_AQ_DROPBOX_ROOT}/Supabase_Backup_db_dump`

Daily output layout:

- `{UK_AQ_DROPBOX_ROOT}/Supabase_Backup_db_dump/ingestdb/YYYY-MM-DD/roles.sql.gz`
- `{UK_AQ_DROPBOX_ROOT}/Supabase_Backup_db_dump/ingestdb/YYYY-MM-DD/schema.sql.gz`
- `{UK_AQ_DROPBOX_ROOT}/Supabase_Backup_db_dump/ingestdb/YYYY-MM-DD/data.sql.gz`
- `{UK_AQ_DROPBOX_ROOT}/Supabase_Backup_db_dump/ingestdb/YYYY-MM-DD/cron_jobs.sql.gz`
- `{UK_AQ_DROPBOX_ROOT}/Supabase_Backup_db_dump/obs_aqidb/YYYY-MM-DD/roles.sql.gz`
- `{UK_AQ_DROPBOX_ROOT}/Supabase_Backup_db_dump/obs_aqidb/YYYY-MM-DD/schema.sql.gz`
- `{UK_AQ_DROPBOX_ROOT}/Supabase_Backup_db_dump/obs_aqidb/YYYY-MM-DD/data.sql.gz`
- `{UK_AQ_DROPBOX_ROOT}/Supabase_Backup_db_dump/obs_aqidb/YYYY-MM-DD/cron_jobs.sql.gz`

Retention is applied separately to the `ingestdb` and `obs_aqidb` dated folders.

Schema dump post-processing notes:

- `schema.sql` is prefixed with `create extension if not exists pg_cron;` when missing.
- For `obs_aqidb`, `schema.sql` is also prefixed with a guarded role-setting block to keep Data API schema exposure stable on restore:
  - `alter role authenticator set pgrst.db_schemas = 'public,graphql_public,uk_aq_public,uk_aq_ops';`
  - the block catches insufficient-privilege / missing-role errors and continues.
- `cron_jobs.sql` is generated from direct `cron.job` reads and uploaded separately, so cron restore does not rely on Supabase CLI including extension-managed schema rows in `data.sql`.

Create or update the daily scheduler job:

```bash
export PROJECT_ID="your-project-id"
export REGION="europe-west2"

bash scripts/gcp/uk_aq_supabase_db_dump_backup_scheduler.sh
```

Default scheduler values:

- cron: `55 0 * * *`
- timezone: `UTC`
- attempt deadline: `1800s`
- max retry attempts: `0`

The scheduler job sends:

- method: `POST`
- URI: `${SERVICE_URL}/run-backup`
- body: `{"trigger_mode":"scheduler"}`
- auth: OIDC using the invoker service account

## Why ingress is internal

This service is intended only for Cloud Scheduler or an authorised internal caller.
Using internal ingress plus IAM-authenticated invocation keeps it off the public unauthenticated surface.

Cloud Scheduler can still invoke the default `run.app` URL when:

- the job is in the same project or VPC Service Controls perimeter
- the Cloud Run service uses internal ingress
- the request is authenticated with OIDC

## Manual test

Because the service is deployed with internal ingress, direct external laptop-to-`run.app`
testing is not the intended path.

Recommended manual test paths:

1. Force the scheduler job to run immediately:

```bash
gcloud scheduler jobs run uk-aq-supabase-db-dump-backup-trigger \
  --project "$PROJECT_ID" \
  --location "$REGION"
```

2. Call `POST /run-backup` from an authorised internal caller inside the same
   project or VPC Service Controls perimeter.

Manual payloads supported by the endpoint:

- both DBs:

```json
{"trigger_mode":"manual"}
```

- `ingestdb` only:

```json
{"trigger_mode":"manual","database":"ingestdb"}
```

- `obs_aqidb` only:

```json
{"trigger_mode":"manual","database":"obs_aqidb"}
```

Scheduled runs always ignore database selection overrides and back up both DBs.

## Restore notes

For each database/day folder:

- `roles.sql.gz`
- `schema.sql.gz`
- `data.sql.gz`

Basic restore flow:

1. download the three files for the database/date
2. decompress:

```bash
gunzip roles.sql.gz
gunzip schema.sql.gz
gunzip data.sql.gz
```

3. restore in order:

```bash
psql "$TARGET_DB_URL" -v ON_ERROR_STOP=1 -f roles.sql
psql "$TARGET_DB_URL" -v ON_ERROR_STOP=1 -f schema.sql
psql "$TARGET_DB_URL" -v ON_ERROR_STOP=1 -f data.sql
```

Restore caveats:

- The Supabase-generated roles dump intentionally omits or comments out reserved Supabase-managed roles and unsafe superuser attributes.
- Restore should be tested against a non-production target first.
- For a managed Supabase target, schema/data restore is usually the more important path; role restore should be reviewed against the destination environment.

## Operational caveats

- Cloud Scheduler HTTP jobs have an attempt-deadline cap, so this service is deployed with a default `1800s` timeout to match that boundary.
- If the total backup duration starts approaching 30 minutes, this service model may need to be split or changed later.
- This implementation still respects the requested Service-based architecture and keeps the workflow simple and operable today.
