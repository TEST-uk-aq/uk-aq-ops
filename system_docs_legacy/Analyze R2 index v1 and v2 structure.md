Executive summary
No code changes were made. I only inspected repository files and the two exported zip folders in _codex_context.

The missing v2 root files are intentional based on current layout docs and generator code. v2 _index_v2 is designed around timeseries file-range indexes only, with two root latest descriptors:

history/_index_v2/observations_timeseries_latest.json

history/_index_v2/aqilevels_hourly_data_timeseries_latest.json
The documented _index_v2 object tree lists only those two latest files and the pollutant-partitioned per-day/per-connector/per-pollutant manifests. 

v2 does not currently define or generate root equivalents of v1 observations_latest.json / aqilevels_latest.json. The v2 generator path builds pollutant index manifests and then writes only the domain-specific v2 timeseries latest file. 

The canonical v2 AQI name is aqilevels_hourly_data_timeseries_latest.json, not aqilevels_timeseries_latest.json. This is reflected in the shared builder, docs, inventory scanner, and integrity tool v2 config. 

The v2 observations and AQI runtime read paths require a pollutant to form the pollutant-partitioned index key. The observations Worker short-circuits v2 reads without pollutant, and the AQI Worker stops v2 scanning with v2_requires_pollutant_partition if pollutant is missing. 

/api/aq/timeseries currently parses and forwards pollutant to the v2 observations R2 Worker. The cache proxy normalizes pollutant, requires it for the R2-first v2 path, and passes it as the pollutant query parameter to the observations R2 Worker. 

I did not find code expecting old v1-style names under _index_v2 such as _index_v2/observations_latest.json, _index_v2/aqilevels_latest.json, or _index_v2/aqilevels_timeseries_latest.json. The remaining references to those v1 names are for history/_index, docs, v1 helpers, or v1 integrity config.

Coverage in the exported files starts at 2026-03-18 for both v2 domains. From the repository alone, that appears to be the exported v2 dataset’s current coverage rather than a schema/layout error. Whether it is an expected backfill cutoff versus incomplete backfill cannot be proven without checking the intended v2 backfill schedule or live R2/source data.

The one-day max-day difference is plausible and likely data/job timing related, not a naming/layout issue: exported v2 observations latest reaches 2026-06-13, while v2 AQI reaches 2026-06-12. Since AQI is derived from observations and the AQI v2 domain is a separate history/v2/aqilevels/hourly/data prefix, a one-day lag can be expected during normal pipeline sequencing, but it should be checked against the daily task health/backfill logs for the specific run.

Whether the two missing v2 root files are intentional
Yes — intentional, based on current repo evidence.

Reasons:

The documented _index_v2 object tree lists only:

observations_timeseries_latest.json

aqilevels_hourly_data_timeseries_latest.json

pollutant-partitioned per-day/per-connector/per-pollutant manifests
It does not list v2 observations_latest.json or aqilevels_latest.json. 

The backup/index docs distinguish v1 and v2 outputs: v1 has four aggregate root index files, while v2 has per-(day, connector, pollutant) manifests and two v2 latest files. 

The v2 index generator writes the v2 timeseries latest descriptor only, using either the observations latest key or AQI hourly data latest key. 

The inventory scanner’s v2 latest file config includes only observations_timeseries_latest.json and aqilevels_hourly_data_timeseries_latest.json. 

Why v2 no longer needs those v1 root files
v2 read acceleration is based on timeseries file-range indexes that are already partitioned by:

day_utc
connector_id
pollutant_code
The v2 latest descriptor summarizes coverage and points to the per-pollutant index key template. 

The actual Workers do not need a domain-level root summary file to resolve reads. They construct the per-day/per-connector/per-pollutant index path directly from the request window, connector, and pollutant.

Risks for the planned R2-first refactor
Pollutant must be present for v2 R2-first reads. If chart or API callers omit pollutant, the cache proxy will not use the observations R2 v2 path and will record pollutant_required_for_v2_r2. 

Any component that assumes v1 AQI name aqilevels_timeseries_latest.json under _index_v2 will fail. Current active code appears aligned to aqilevels_hourly_data_timeseries_latest.json, but external scripts/runbooks should be checked.

The v2 coverage window is much shorter than v1. R2-first use over historical windows before 2026-03-18 must fall back to v1/Supabase/other source unless v2 is backfilled further.

Observations and AQI v2 max days can diverge. The current export shows observations one day ahead of AQI. That is plausible for a derived AQI pipeline, but the R2-first refactor should treat AQI and observations coverage independently.

Do not introduce root v2 summary files unless a consumer actually needs them. Adding nonessential v2 root files could create confusion with v1 semantics and increase inventory/backup surface area.
